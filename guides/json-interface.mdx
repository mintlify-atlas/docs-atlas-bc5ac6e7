The JSON interface exposes TDLib as a set of C functions that accept and return null-terminated JSON strings. Any language that can call C functions and parse JSON can use TDLib through this interface — no language-specific binding required.

<Note>
  The JSON interface follows semantic versioning. Libraries with the same major version number are binary and backward compatible. Use the `version` option value to detect the exact TDLib version at runtime and select appropriate API methods.
</Note>

## The four core functions

All interaction with TDLib over the JSON interface flows through four C functions declared in `td/telegram/td_json_client.h`:

| Function | Signature | Description |
|---|---|---|
| `td_create_client_id` | `int td_create_client_id()` | Creates a new TDLib client instance and returns its opaque integer identifier. The instance does not send updates until the first request is sent to it. |
| `td_send` | `void td_send(int client_id, const char *request)` | Sends an asynchronous JSON request to the given client. May be called from any thread. |
| `td_receive` | `const char *td_receive(double timeout)` | Blocks for up to `timeout` seconds and returns the next available update or response across all clients as a JSON string, or `NULL` on timeout. Must not be called from two threads simultaneously. |
| `td_execute` | `const char *td_execute(const char *request)` | Executes a request synchronously. Only requests documented as "Can be called synchronously" are supported. |

The returned pointer from `td_receive` and `td_execute` is valid only until the next call to either function.

## The `@type` and `@extra` fields

Every JSON object exchanged with TDLib uses two special fields:

- **`@type`** — identifies the TDLib API object or function. Required on all requests; present on all responses. Example: `"@type": "getMe"`.
- **`@extra`** — an optional value you attach to a request. TDLib echoes the same value back in the corresponding response, letting you match responses to requests. Every response also includes `"@client_id"` with the identifier of the originating client.

Updates pushed by TDLib (for example `updateAuthorizationState`) carry only `@type` and `@client_id`; they have no `@extra`.

## General usage pattern

<Steps>
  <Step title="Load the shared library">
    Load `tdjson` (or `tdjson.dll` on Windows) using your language's FFI mechanism and declare the four function signatures.
  </Step>
  <Step title="Create a client">
    Call `td_create_client_id()` to obtain a client identifier. Share this value with any threads that need to send requests.
  </Step>
  <Step title="Activate the client">
    Send at least one request before calling `td_receive`. A lightweight request such as `getOption` with `name: "version"` is conventional.
  </Step>
  <Step title="Run the receive loop">
    Loop on `td_receive(timeout)`. When a non-NULL result arrives, parse it as JSON and dispatch on `@type`. Responses to your requests carry the `@extra` value you supplied; updates have `request_id == 0`.
  </Step>
  <Step title="Handle authorization">
    Respond to `updateAuthorizationState` updates by sending the appropriate requests (`setTdlibParameters`, `setAuthenticationPhoneNumber`, etc.). See [authorization](/concepts/authorization) for the full state machine.
  </Step>
</Steps>

```c event loop skeleton
int client_id = td_create_client_id();

// activate the client
td_send(client_id, "{\"@type\":\"getOption\", \"name\":\"version\"}");

const double WAIT_TIMEOUT = 10.0;
while (true) {
  const char *result = td_receive(WAIT_TIMEOUT);
  if (result != NULL) {
    // parse result as JSON, dispatch on @type
  }
}
```

## Python example

The following example is adapted from `example/python/tdjson_example.py`. It uses `ctypes` to load `libtdjson` and wraps the four core functions.

```python loading the library and declaring function signatures
from ctypes import CDLL, CFUNCTYPE, c_char_p, c_double, c_int
from ctypes.util import find_library
import json, os, sys

tdjson_path = find_library("tdjson")
if tdjson_path is None:
    if os.name == "nt":
        tdjson_path = os.path.join(os.path.dirname(__file__), "tdjson.dll")
    else:
        sys.exit("Error: Can't find 'tdjson' library.")

tdjson = CDLL(tdjson_path)

_td_create_client_id = tdjson.td_create_client_id
_td_create_client_id.restype = c_int
_td_create_client_id.argtypes = []

_td_receive = tdjson.td_receive
_td_receive.restype = c_char_p
_td_receive.argtypes = [c_double]

_td_send = tdjson.td_send
_td_send.restype = None
_td_send.argtypes = [c_int, c_char_p]

_td_execute = tdjson.td_execute
_td_execute.restype = c_char_p
_td_execute.argtypes = [c_char_p]
```

```python helper wrappers
def execute(query):
    query_json = json.dumps(query).encode("utf-8")
    result = _td_execute(query_json)
    if result:
        return json.loads(result.decode("utf-8"))
    return None

def send(client_id, query):
    query_json = json.dumps(query).encode("utf-8")
    _td_send(client_id, query_json)

def receive(timeout=1.0):
    result = _td_receive(timeout)
    if result:
        return json.loads(result.decode("utf-8"))
    return None
```

```python creating a client and running the event loop
# set log verbosity before creating the client
execute({"@type": "setLogVerbosityLevel", "new_verbosity_level": 1})

client_id = _td_create_client_id()

# activate the client
send(client_id, {"@type": "getOption", "name": "version"})

while True:
    event = receive()
    if event:
        print(json.dumps(event, indent=2))
```

```python handling authorizationStateWaitTdlibParameters
if event["@type"] == "updateAuthorizationState":
    auth_type = event["authorization_state"]["@type"]

    if auth_type == "authorizationStateWaitTdlibParameters":
        send(client_id, {
            "@type": "setTdlibParameters",
            "database_directory": "tdlib_data",
            "use_message_database": True,
            "use_secret_chats": True,
            "api_id": YOUR_API_ID,       # obtain from https://my.telegram.org
            "api_hash": YOUR_API_HASH,
            "system_language_code": "en",
            "device_model": "Python TDLib Client",
            "application_version": "1.0",
        })
```

<Warning>
  Always obtain your own `api_id` and `api_hash` from [https://my.telegram.org](https://my.telegram.org). Do not ship example credentials in production applications.
</Warning>

## Synchronous execution

A small set of requests can be executed without a running client by calling `td_execute` directly. These requests are documented with "Can be called synchronously" in the API reference. A common example is `setLogVerbosityLevel`:

```python synchronous execute
result = execute({"@type": "setLogVerbosityLevel", "new_verbosity_level": 1})
```

```python synchronous text entity parsing
result = execute({
    "@type": "getTextEntities",
    "text": "@telegram /test_command https://telegram.org telegram.me",
})
```

## Browser and WebAssembly

For browser environments, use [tdweb](https://github.com/tdlib/td/tree/master/example/web) — an official wrapper that compiles TDLib to WebAssembly and provides a JavaScript API. See [telegram-react](https://github.com/evgeny-nadymov/telegram-react) for a complete example application built on tdweb.

## Old JSON interface

`td_json_client.h` also documents an older per-client API (`td_json_client_create`, `td_json_client_send`, `td_json_client_receive`, `td_json_client_destroy`). This interface will be removed in TDLib 2.0.0. Prefer the newer `td_create_client_id` / `td_send` / `td_receive` API for all new integrations.
